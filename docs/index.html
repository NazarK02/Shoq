// Core features

Built with the right primitives.

Every design decision prioritises correctness over convenience. No homegrown crypto, no shortcuts, no marketing claims that can't be verified.

🔒
XChaCha20-Poly1305 Encryption
Messages are encrypted client-side using XChaCha20-Poly1305 via libsodium before leaving your device. Firebase stores only authenticated ciphertext.
libsodium
🔑
X25519 Key Exchange
Session keys are derived from ephemeral X25519 Diffie-Hellman. The server never has access to private keys or shared secrets at any point.
X25519
🪪
Ed25519 Identity Keys
Each user has a long-term Ed25519 signing keypair. Identity is tied to Firebase Auth UID. Public keys are published to Firestore, private keys never leave the device.
identity
📲
Silent Push Notifications
FCM notifications contain zero message content. Payloads carry only metadata needed to trigger a local fetch. E2EE guarantees are never weakened by notification design.
FCM
🗄️
Firestore as Encrypted Transport
Firestore is treated as an untrusted relay. All documents written to it are ciphertext. Firestore rules enforce access without needing to read content.
zero-trust
🔐
Platform Secure Storage
Private keys are stored in platform-native secure enclaves: Keychain (iOS/macOS), Keystore (Android), DPAPI (Windows), SecretService (Linux).
per-platform
🔄
Key Rotation
Signed prekey rotation following a Signal-inspired model. Rotating keys invalidate old sessions without disrupting existing contacts. Rotation events are logged and verifiable.
PFS
📱
Multi-device Support
Each device maintains its own keypair. New device enrollment requires explicit approval from an existing device. No silent device addition is possible.
devices

Room & Channel Architecture
Servers (rooms) host named channels: text, voice, file, forum, and assignment channels. Chat folders organise direct conversations. All message content is encrypted regardless of channel type.

// Architecture

How messages flow through the system.

Encryption and decryption occur exclusively on client devices. Firebase is used as an authenticated, encrypted transport layer — not as a trusted service.

// Encryption layer

Clean separation of concerns.

The crypto service is isolated, dependency-injected, and never touches UI state. Firestore sees only opaque bytes.

lib/services/crypto_service.dart
/// Encrypt a plaintext message for a given recipient public key.
/// Returns base64-encoded ciphertext safe to store in Firestore.
Future<String> encryptMessage({
  required String plaintext,
  required Uint8List recipientPublicKey,
  required Uint8List senderPrivateKey,
}) async {
  // Derive shared secret via X25519
  final sharedSecret = _sodium.crypto.box.beforenm(
    theirPublicKey: recipientPublicKey,
    mySecretKey: SecureKey.fromList(_sodium, senderPrivateKey),
  );

  // Generate fresh nonce for every message
  final nonce = _sodium.randombytes.buf(
    _sodium.crypto.secretBox.nonceBytes,
  );

  final ciphertext = _sodium.crypto.secretBox.easy(
    message: Uint8List.fromList(utf8.encode(plaintext)),
    nonce: nonce,
    key: sharedSecret,
  );

  // Store nonce alongside ciphertext — it is not secret
  return base64Url.encode([...nonce, ...ciphertext]);
}
// Security model

Honest about what it protects — and what it does not.

Security claims are only made where they can be demonstrated. Limitations are documented explicitly.

Protected
Message content confidentiality
Firestore never receives plaintext. Encryption happens before any network call. Even Anthropic or Firebase employees cannot read your messages.
Protected
Private key isolation
Private keys are stored exclusively in platform secure storage and never transmitted. A compromised Firebase account does not expose private keys.
Protected
Push notification content
FCM payloads contain no message body, no sender identity, no subject. Notifications are opaque triggers that instruct the client to fetch and decrypt locally.
Protected
Forward secrecy (per-session)
Ephemeral X25519 keys are used per session. Compromise of a long-term identity key does not retroactively decrypt past sessions.
Visible to server
Metadata (who talks to whom)
Firestore document paths reveal conversation participants by UID. Timestamps and message counts are visible. This is a known limitation, not a hidden risk.
Out of scope
Endpoint device security
If the device itself is compromised (malware, physical access, screen capture), E2EE cannot protect content at the point of display. This is an inherent limitation of all E2EE systems.
