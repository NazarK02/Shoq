Architecture

Firebase is the transport. Clients own encryption work.

Message bodies are encrypted before Firestore writes. The chat service builds per-recipient ciphertext maps, stores sender and recipient routing metadata, and lets clients decrypt locally with their device key material.

Security status

Honest claims only.

Shoq has real client-side encrypted message bodies, but it is not ready for high-risk use. The current documentation intentionally names the gaps that must be fixed before production security claims.

Implemented

Encrypted chat bodies

Text, sticker, location, and room message bodies are encrypted client-side with libsodium before Firestore writes.

Implemented

Per-device ciphertext maps

Messages can include recipient maps keyed by user/device and short public-key identifiers for multi-device delivery.

Visible

Metadata is not protected

Firebase can observe participants, timestamps, message counts, room membership, device counts, notification routing, and status.

High priority

No key verification

There is no fingerprint or safety-number comparison yet, so public-key substitution remains possible.

High priority

Key backup needs redesign

The current migration/recovery path can store account key material in Firestore. This weakens the server-compromise model.

High priority

Files are not app-layer encrypted

Firebase Storage file bytes are not currently encrypted by Shoq before upload. Some file metadata can also remain plaintext.

Current crypto stack

Message body encryption: libsodium crypto.box.easy/openEasy
Key agreement:           Curve25519/X25519-compatible crypto.box keypairs
Cipher construction:     libsodium crypto_box, XSalsa20-Poly1305
Public key IDs:          SHA-256 hash prefix of base64 public key
Local storage:           flutter_secure_storage

Not currently implemented:
- Ed25519 identity signatures
- XChaCha20-Poly1305 chat message encryption
- signed prekey rotation
- fingerprint or safety-number verification
- app-layer encrypted file bytes
